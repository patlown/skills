# Keep Moving

When uncertainty remains, keep useful work moving without pretending the uncertainty is gone.

## Rules

1. Inspect what can be inspected before asking.
2. Ask only when a wrong assumption would cause meaningful rework or commit to unclear intent.
3. Make the smallest explicit assumption that preserves momentum.
4. Prefer reversible progress: investigation, tests, prototypes, drafts, or scoped code changes.
5. Surface unresolved questions when they matter.
6. Stop before irreversible data changes, external side effects, broad architecture commitments, or product decisions the user has not actually made.

## Implementation Under Uncertainty

Implementation is allowed when the assumption is small, evidence-backed, and easy to revise.

Do not use uncertainty as an excuse for speculative design. Move on the part that is clear, then re-evaluate.
